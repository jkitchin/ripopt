# V0.8 Data-Layout Refactor: ripopt → Ipopt 3.14 split storage

Status: Phase 0 complete (this document). Phases 1-5 pending.

## Motivation

ripopt currently stores constraints as a single combined `g(x)` vector with
per-row `g_l[i]` / `g_u[i]`, and a `ConstraintPartition` (re)derived from the
bounds at every assemble. Equality rows carry sentinel slacks, inequality
rows carry real ones, and the `ConstraintPartition::eq_pos` /
`ineq_pos` indirection is consulted ~5× per iteration to route data into the
4-block KKT system.

Ipopt 3.14, by contrast, splits `c(x)` and `d(x)` at the TNLP adapter and the
IPM core never sees a combined view. Every iter-0 alignment probe on
arki0003 has had to *recompute* the c/d split inside ripopt to compare
against Ipopt's `delta[2][k_c]` / `delta[3][k_d]` — and we still have a
6.16% dx[1753] gap and a 13.1% dy[e2253] gap (at an unscaled equality) that
local-system inspection has not closed (`docs/A8_FOLLOWUP_arki0003.md`
A8.21–A8.22).

This refactor rewrites the IPM-internal data structures to match Ipopt's
split layout exactly. The `NlpProblem` trait and the NL/AMPL parser
boundary remain combined — the split is a pure IPM-internal concept.

### ripopt-side call-site enumeration

Verified by code survey at HEAD (commit `04917eb`):

| Pattern | Hits |
|---|---|
| `state.g\b` | 55 |
| `state.g_l` | 100 |
| `state.g_u` | 91 |
| `state.s\b` | 63 |
| `state.y\b` | 58 |
| `state.jac_*` | 41 |
| `ConstraintPartition::new` | 16 (5 prod, 11 test) |

Concentrated in `src/ipm.rs`. Cross-module sites (`convergence.rs`,
`filter.rs`, `kkt.rs`, `kkt_aug.rs`, `restoration*.rs`) consume **bare
slices** — they don't read `state.*` directly, so the migration there is
signature-level.

`g_scaling` is duplicated on `ScaledProblem` (`src/ipm.rs:186`) **and**
`SolverState` (`src/ipm.rs:669`). Both will be replaced by `c_scaling` /
`d_scaling`.

The NL parser (`src/nl/parser.rs:62-69, 155-156, 278-284`) and the C-API
(`src/c_api.rs:866-880`) populate raw combined `g_l`/`g_u` and Jacobian
col-ptrs. **The trait boundary stays combined.** The split happens once
when `SolverState` ingests the problem.

## Mapping (today → target)

| Concept | Today | Target | Sizing |
|---|---|---|---|
| `state.g, g_l, g_u` (size m) | flat `Vec<f64>` | `g_c, g_d, d_l, d_u` | n_c + n_d (n_d each) |
| `state.s` (size m, eq=sentinel) | flat | `s` (d-block only) | n_d |
| `state.y, dy` (size m) | flat | `y_c, y_d, dy_c, dy_d` | n_c + n_d |
| `state.v_l, v_u` (size m, eq=0) | flat | `v_l, v_u` (d-block only, sized to # finite d bounds via Pd_L/Pd_U) | depends on bound counts |
| `state.g_scaling` (size m) | flat (also on ScaledProblem) | `c_scaling, d_scaling` (single home: split layout on state) | n_c + n_d |
| Jacobian | one combined-row triplet `(jac_rows, jac_cols, jac_vals)` | `Jac_c` (n_c × n) + `Jac_d` (n_d × n), each its own triplet | by block |
| `ConstraintPartition` | rebuilt at every assemble (16 call sites) | one `ConstraintLayout` on state, built once per problem | — |

Note: ripopt today does not have `Px_L/Px_U` expansion matrices for variable
bounds either — `z_l/z_u` are sized `n` with zero entries on infinite-bound
indices. We keep that for now (out of scope for this refactor; it's a
separate v0.9 concern). Likewise we initially keep `v_l/v_u` sized `n_d`
with zeros on infinite-bound rows, rather than introducing `Pd_L/Pd_U`.

## Phased plan

### Phase 1 — `ConstraintLayout` + accessors (no storage change)

Goal: introduce the type system without touching call sites.

1. New module `src/constraint_layout.rs` (or `src/ipm/layout.rs`) with:
   ```rust
   pub struct ConstraintLayout {
       pub n_c: usize,
       pub n_d: usize,
       pub eq_pos: Vec<Option<usize>>,    // m → Some(k_c) for equalities
       pub ineq_pos: Vec<Option<usize>>,  // m → Some(k_d) for inequalities
       pub c_to_combined: Vec<usize>,     // n_c → combined index (the c_map)
       pub d_to_combined: Vec<usize>,     // n_d → combined index (the d_map)
   }
   ```
2. Add `pub layout: ConstraintLayout` field to `SolverState`. Build it once
   in `SolverState::new` from `g_l`/`g_u`. Combined storage stays.
3. Add accessor methods on `SolverState` returning `Vec<f64>` *copies*
   (Phase 1 — slow but correct). Phase 3 swaps them to bare-slice views:
   - `g_c() -> Vec<f64>`, `g_d() -> Vec<f64>`
   - `d_l() -> Vec<f64>`, `d_u() -> Vec<f64>`
   - `c_scaling() -> Vec<f64>`, `d_scaling() -> Vec<f64>`
   - `y_c() -> Vec<f64>`, `y_d() -> Vec<f64>`
   - `s_d() -> Vec<f64>` (drops eq sentinels)
4. Refactor `ConstraintPartition` to be a thin façade over
   `ConstraintLayout` (or replace at all 5 prod sites; tests retained).

Verification: `cargo test` (lib + integration) green. No behaviour change.
Single commit.

### Phase 2 — migrate read sites to accessors module-by-module

Order chosen by isolation (cross-module sites first — they're already
slice-based, so the change is just call-site rewiring):

1. **convergence.rs** — `compute_residuals` and friends already take slices.
   Update `src/ipm.rs` callers to pass `state.g_c()` / `state.g_d()` /
   `state.d_l()` / `state.d_u()` / `state.y_c()` / `state.y_d()` instead of
   the combined slices. Internal logic in `convergence.rs` switches from
   `is_equality_constraint(g_l[i], g_u[i])` branching to direct c/d-block
   loops.
2. **filter.rs** — `theta` (constraint violation) split into
   `theta_c + theta_d`. Equality contribution is `||g_c||`, inequality is
   `||max(d_l - d, 0) + max(d - d_u, 0)||`.
3. **kkt_aug.rs** — `assemble_aug_kkt`, `build_outer_rhs`, `unpack_step`,
   `compute_sigma_s`. The 4-block KKT structure already matches; the
   change here is feeding `Jac_c` / `Jac_d` views (still combined-backed)
   instead of `(jac_rows, jac_cols, jac_vals) + partition.eq_pos`.
4. **kkt.rs** — full-matrix variant; same migration as kkt_aug.
5. **restoration*.rs** — `RestorationNlp` exposes its own combined `g`. We
   build the layout for the restoration NLP independently in
   `RestorationNlp::new`. Out of scope for this phase to split *its*
   storage — the wrapper just needs to not assume the parent uses combined.
6. **ipm.rs** — line search, watchdog, SOC, iter-0 probes, result
   unscaling. ~430 hits; tackle one bucket per commit.

Verification: `cargo test` green per commit. Spot-check `cargo run --bin
cutest_suite -- arki0003` to confirm iter-0 numbers unchanged (this is
Phase 2's invariant — pure refactor).

#### Phase 2 status (2026-04-30)

Steps 1–5 are landed (commits 81c331e, 516aea1, 6fd01af, 826d131).
Filter (step 2) was a no-op: filter.rs is scalar-only (theta/phi);
the layout-aware migration happened in its callers in step 1.

Step 6 (ipm.rs internal sweep) is split:
* **Pass-throughs** (~250 reads) like `&state.g_l, &state.g_u` feeding
  `kkt::assemble_kkt` / `kkt_aug::*` / `compute_sigma_s` etc. are kept
  as-is — those leaf functions are already layout-driven (steps 3, 4)
  so no rewire is needed until Phase 3 flips the storage shape.
* **Inline equality checks** at 12 sites of the form
  `state.g_l[i].is_finite() && state.g_u[i].is_finite() &&
   (state.g_l[i] - state.g_u[i]).abs() < 1e-14` use a 1e-14 threshold,
  whereas `ConstraintLayout` (and Ipopt) use 1e-15. Replacing them is
  a minor numeric change — not a pure refactor — and is deferred to a
  separate hypothesis-driven commit so any benchmark delta is
  attributable to that change in isolation.
* **Specific-row reads** (`state.y[i]`, `state.g[i]` at single indices
  inside line-search / SOC / iter-0 probes) are subsumed by Phase 3:
  flipping the storage means there is no combined `state.y` to read
  — the migration becomes mechanical at that point.

The Phase 2 invariant ("no behaviour change, leaf modules layout-aware")
is met. 303 lib tests + all integration suites pass; the only failing
test (electrolyte_02_co2_water) is pre-existing on baseline.

### Phase 3 — flip backing storage to split

After Phase 2, no caller of `state.g`/`g_l`/`g_u`/`s`/`y`/`g_scaling`
remains outside the accessors. We can now swap the fields:

```rust
// Before (today, ~src/ipm.rs:625-669):
pub g: Vec<f64>,
pub g_l: Vec<f64>,
pub g_u: Vec<f64>,
pub s: Vec<f64>,
pub y: Vec<f64>,
pub v_l: Vec<f64>,
pub v_u: Vec<f64>,
pub g_scaling: Vec<f64>,

// After:
pub g_c: Vec<f64>,
pub g_d: Vec<f64>,
pub d_l: Vec<f64>,
pub d_u: Vec<f64>,
pub s: Vec<f64>,    // size n_d only
pub y_c: Vec<f64>,
pub y_d: Vec<f64>,
pub v_l: Vec<f64>,  // size n_d
pub v_u: Vec<f64>,  // size n_d
pub c_scaling: Vec<f64>,
pub d_scaling: Vec<f64>,
```

Update `SolverState::new`, `ScaledProblem::constraint_bounds` /
`constraints` / `jacobian_values` to populate the split storage directly.
Accessors become trivial pass-throughs (and switch from returning `Vec` to
returning `&[f64]`).

Verification: `cargo test` green. `cargo run --bin cutest_suite --
arki0003` numerics unchanged. Result-unscaling code (`src/ipm.rs:8811,
8815`) updated to reconstruct user-ordered `g`/`y` via
`layout.c_to_combined`/`d_to_combined`.

3f landed as additive (split fields populated alongside combined
`g_l`/`g_u`) rather than a full storage flip. Rationale: 125+ read
sites of `state.g_l` / `state.g_u` make a mass mechanical replace
high-risk for a refactor expected to be numerically inert. Bounds
don't change during solve, so the ~3·m extra floats are one-shot
overhead. The native split fields exist on `SolverState` (`c_rhs`,
`d_l`, `d_u`) and Phase 4 readers can migrate incrementally to read
them once a layout-aware bound setter is added to the test-fixture
surface.

`state.g` (constraint values) was left as combined m-form scratch.
It is the natural target for the `eval_g` trait boundary, and
splitting it would require a transient `g_eval_buf` plus per-eval
projection — overhead with no Phase 4 benefit (Phase 4 only consumes
the Jacobian split, not the constraint-value split).

**Phase 3 complete.** Phase 4 (Jacobian split) is unblocked.

### Phase 4 — split Jacobian into Jac_c, Jac_d

Replace combined `(jac_rows, jac_cols, jac_vals)` with two sparse blocks.
`ScaledProblem::jacobian_values` populates them directly. `kkt_aug` no
longer consults `partition.eq_pos`/`ineq_pos` per triplet — the split is
already done at storage time, matching Ipopt.

`ConstraintPartition` either becomes a no-op shim (kept temporarily for
test compatibility) or is deleted outright (rewriting the 11 tests).

Verification: `cargo test` green. Iter-0 dx/dy/ds on arki0003 must remain
*bit-identical* to Phase 3 output (refactor invariant). If they shift,
that's a Phase 4 bug, not the original gap.

### Phase 6 — compress bound multipliers `z_L`, `z_U` (introduce `Px_L`, `Px_U`)

Today: `state.z_l`, `state.z_u` are size `n` with `0` for unbounded sides.
Ipopt: `z_L` is size `n_x_l` (count of finite *lower* bounds), recovered
to full-n via the expansion matrix `Px_L` (`IpOrigIpoptNLP.hpp:197`).

- 6a. Add `state.bound_layout`: counts `n_x_l`, `n_x_u` plus index maps

  Ipopt's `Px_L_` ExpansionMatrix.
- 6b. Add native compressed `z_l_compressed: Vec<f64>` (size n_x_l),
  `z_u_compressed: Vec<f64>` (size n_x_u). Populate as projections of
  the combined `z_l`/`z_u` (additive).
- 6c. Migrate readers (sigma_x compute, rhs_z_l/u, dz_l/u recovery,
  frac-to-boundary, kappa_sigma reset, LOQO update) to consume the

- 6d. Remove combined `z_l`, `z_u` from `SolverState`.

### Phase 10 — physical row reorder at the IPM boundary ✅ COMPLETE

Ipopt's `TNLPAdapter` returns the user-supplied `g(x)` and `jac_g`, then
*physically* projects into split form using `c_map`/`d_map`. After
TNLPAdapter, no combined m-vector exists.

ripopt today preserves the user's row order in the combined-form fields
that survived earlier phases. After Phases 5–9, the only remaining user-
row-order surface is the `NlpProblem` trait itself (`fn constraints`,
`fn jacobian_*`). This phase formalises the boundary:

- 10a ✅. `src/split_nlp.rs` (`c317387`) — internal `SplitNlp<'a, P>`
  adapter wraps the user's `NlpProblem` and a per-solve
  `ConstraintLayout`, exposing split-form `constraints_split` /
  `jacobian_split` (which project into `c`/`d` and `Jac_c`/`Jac_d` via
  the layout's row maps) plus combined-form `_combined` pass-throughs
  for the few sites that still need m-form residuals.
- 10b ✅. `92fa2e5` (10b.1) + `98423be` (10b.2-5) — every
  post-`SolverState::new` `problem.X` callsite in `ipm.rs` routes
  through the adapter:
    * `evaluate_with_linear` (canonical iter eval)
    * `evaluate_soc_trial_and_check` (SOC trial eval)
    * `evaluate_trial_point` (line-search backtracking)
    * watchdog post-restoration filter gate + original-NLP eval
    * `compute_min_dual_infeas_alpha` (post-step Lagrangian re-eval)
    * GN-restoration entry-point closures (both code paths)
  The remaining `problem.X` callsites in `ipm.rs` (structure queries
  in `SolverState::new`, scaling preflight, LS y-init) all run
  *upstream* of the boundary, matching Ipopt where TNLPAdapter's
  setup phase queries the user TNLP directly.
- 10c ✅. The `NlpProblem` trait surface is unchanged. No user-visible
  API change. `src/restoration_nlp.rs` is a TNLPAdapter-side
  *implementation* of `NlpProblem` (the resto NLP wraps the original
  problem and implements the user trait), not a consumer, so it has
  no boundary sites to migrate. `src/sensitivity.rs` and
  `src/linearity.rs` (post-solve / preprocessing diagnostics) and
  `src/lbfgs.rs` (alternative L-BFGS solver path) sit outside the
  IPM main driver and remain on direct `problem.X` calls — they have
  their own boundaries.

### Phase 11 — fixed-variable strategy (Ipopt `MAKE_PARAMETER`) ✅ COMPLETE

**Architectural finding (2026-05-02):** ripopt already implements
`MAKE_PARAMETER` semantics — the elimination just lives at the
preprocessor layer (`PreprocessedProblem::new_fixed_only`,
`src/preprocessing.rs:58`) rather than at an IPM-internal `x_layout`.
The wrapper presents the IPM with an NLP whose `n` is already
`n_full − n_fixed` and whose Jacobian/Hessian columns are already
remapped, exactly mirroring what Ipopt's `TNLPAdapter` does between
the user `TNLP` and `OrigIpoptNLP`. The IPM main driver therefore
needs no internal x-compression; it sees a reduced NLP from the start.

This makes the originally-planned 11a/11b/11c (introduce `x_layout`,
compress `x`/`x_l`/`x_u`/`z_l`/`z_u`/`grad_f`/Hessian rows·cols,
Jacobian cols, switch handling) unnecessary work. ripopt's split is
"preprocessor wraps the user NLP / IPM operates on the reduced NLP",
where Ipopt's split is "TNLPAdapter wraps the user TNLP / IpoptNLP
operates on the reduced TNLP". Same boundary, different layer name.

Phase 11 deliverable becomes a default-flip + documentation closeout:

- ✅ Flip `FixedVariableTreatment::default()` from `RelaxBounds` to
  `MakeParameter` (`src/options.rs`), aligning with Ipopt 3.14's
  `fixed_variable_treatment = make_parameter` default.
- ✅ Document the preprocessor-layer architecture in the
  `FixedVariableTreatment` rustdoc.
- ✅ Update `test_relax_fixed_variable_bounds_default_widens` to
  reflect the new default — the IPM-layer `relax_fixed_variable_bounds`
  is retained as a safety net for callers that bypass the preprocessor
  and reach the IPM with literally-equal bounds (its widening branch
  fires for both `RelaxBounds` and `MakeParameter`, making the test
  pass under either default).

`RelaxBounds` remains as the legacy treatment for callers that don't
want preprocessor-layer elimination (e.g. when they need
identity-mapped variable indices through to the solution).

`relax_fixed_variable_bounds` (the IPM-layer ±1e-8·max(|c|,1) widening
function) is **not** removed — it's the IPM's safety-net widening
when the preprocessor was bypassed. It is *not* synonymous with the
`RelaxBounds` enum variant; that's a Phase-11 naming clarification
captured in the rustdoc.

323/323 lib tests pass with `MakeParameter` as the default.
