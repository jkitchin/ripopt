# V0.8 Data-Layout Refactor: ripopt ‚Üí Ipopt 3.14 split storage

Status: Phase 0 complete (this document). Phases 1-5 pending.

## Motivation

ripopt currently stores constraints as a single combined `g(x)` vector with
per-row `g_l[i]` / `g_u[i]`, and a `ConstraintPartition` (re)derived from the
bounds at every assemble. Equality rows carry sentinel slacks, inequality
rows carry real ones, and the `ConstraintPartition::eq_pos` /
`ineq_pos` indirection is consulted ~5√ó per iteration to route data into the
4-block KKT system.

Ipopt 3.14, by contrast, splits `c(x)` and `d(x)` at the TNLP adapter and the
IPM core never sees a combined view. Every iter-0 alignment probe on
arki0003 has had to *recompute* the c/d split inside ripopt to compare
against Ipopt's `delta[2][k_c]` / `delta[3][k_d]` ‚Äî and we still have a
6.16% dx[1753] gap and a 13.1% dy[e2253] gap (at an unscaled equality) that
local-system inspection has not closed (`docs/A8_FOLLOWUP_arki0003.md`
A8.21‚ÄìA8.22).

This refactor rewrites the IPM-internal data structures to match Ipopt's
split layout exactly. The `NlpProblem` trait and the NL/AMPL parser
boundary remain combined ‚Äî the split is a pure IPM-internal concept.

### ripopt-side call-site enumeration

Verified by code survey at HEAD (commit `04917eb`):

| Pattern | Hits |
|---|---|
| `state.g\b` | 55 |
| `state.g_l` | 100 |
| `state.g_u` | 91 |
| `state.s\b` | 63 |
| `state.y\b` | 58 |
| `state.jac_*` | 41 |
| `ConstraintPartition::new` | 16 (5 prod, 11 test) |

Concentrated in `src/ipm.rs`. Cross-module sites (`convergence.rs`,
`filter.rs`, `kkt.rs`, `kkt_aug.rs`, `restoration*.rs`) consume **bare
slices** ‚Äî they don't read `state.*` directly, so the migration there is
signature-level.

`g_scaling` is duplicated on `ScaledProblem` (`src/ipm.rs:186`) **and**
`SolverState` (`src/ipm.rs:669`). Both will be replaced by `c_scaling` /
`d_scaling`.

The NL parser (`src/nl/parser.rs:62-69, 155-156, 278-284`) and the C-API
(`src/c_api.rs:866-880`) populate raw combined `g_l`/`g_u` and Jacobian
col-ptrs. **The trait boundary stays combined.** The split happens once
when `SolverState` ingests the problem.

## Mapping (today ‚Üí target)

| Concept | Today | Target | Sizing |
|---|---|---|---|
| `state.g, g_l, g_u` (size m) | flat `Vec<f64>` | `g_c, g_d, d_l, d_u` | n_c + n_d (n_d each) |
| `state.s` (size m, eq=sentinel) | flat | `s` (d-block only) | n_d |
| `state.y, dy` (size m) | flat | `y_c, y_d, dy_c, dy_d` | n_c + n_d |
| `state.v_l, v_u` (size m, eq=0) | flat | `v_l, v_u` (d-block only, sized to # finite d bounds via Pd_L/Pd_U) | depends on bound counts |
| `state.g_scaling` (size m) | flat (also on ScaledProblem) | `c_scaling, d_scaling` (single home: split layout on state) | n_c + n_d |
| Jacobian | one combined-row triplet `(jac_rows, jac_cols, jac_vals)` | `Jac_c` (n_c √ó n) + `Jac_d` (n_d √ó n), each its own triplet | by block |
| `ConstraintPartition` | rebuilt at every assemble (16 call sites) | one `ConstraintLayout` on state, built once per problem | ‚Äî |

Note: ripopt today does not have `Px_L/Px_U` expansion matrices for variable
bounds either ‚Äî `z_l/z_u` are sized `n` with zero entries on infinite-bound
indices. We keep that for now (out of scope for this refactor; it's a
separate v0.9 concern). Likewise we initially keep `v_l/v_u` sized `n_d`
with zeros on infinite-bound rows, rather than introducing `Pd_L/Pd_U`.

## Phased plan

### Phase 1 ‚Äî `ConstraintLayout` + accessors (no storage change)

Goal: introduce the type system without touching call sites.

1. New module `src/constraint_layout.rs` (or `src/ipm/layout.rs`) with:
   ```rust
   pub struct ConstraintLayout {
       pub n_c: usize,
       pub n_d: usize,
       pub eq_pos: Vec<Option<usize>>,    // m ‚Üí Some(k_c) for equalities
       pub ineq_pos: Vec<Option<usize>>,  // m ‚Üí Some(k_d) for inequalities
       pub c_to_combined: Vec<usize>,     // n_c ‚Üí combined index (the c_map)
       pub d_to_combined: Vec<usize>,     // n_d ‚Üí combined index (the d_map)
   }
   ```
2. Add `pub layout: ConstraintLayout` field to `SolverState`. Build it once
   in `SolverState::new` from `g_l`/`g_u`. Combined storage stays.
3. Add accessor methods on `SolverState` returning `Vec<f64>` *copies*
   (Phase 1 ‚Äî slow but correct). Phase 3 swaps them to bare-slice views:
   - `g_c() -> Vec<f64>`, `g_d() -> Vec<f64>`
   - `d_l() -> Vec<f64>`, `d_u() -> Vec<f64>`
   - `c_scaling() -> Vec<f64>`, `d_scaling() -> Vec<f64>`
   - `y_c() -> Vec<f64>`, `y_d() -> Vec<f64>`
   - `s_d() -> Vec<f64>` (drops eq sentinels)
4. Refactor `ConstraintPartition` to be a thin fa√ßade over
   `ConstraintLayout` (or replace at all 5 prod sites; tests retained).

Verification: `cargo test` (lib + integration) green. No behaviour change.
Single commit.

### Phase 2 ‚Äî migrate read sites to accessors module-by-module

Order chosen by isolation (cross-module sites first ‚Äî they're already
slice-based, so the change is just call-site rewiring):

1. **convergence.rs** ‚Äî `compute_residuals` and friends already take slices.
   Update `src/ipm.rs` callers to pass `state.g_c()` / `state.g_d()` /
   `state.d_l()` / `state.d_u()` / `state.y_c()` / `state.y_d()` instead of
   the combined slices. Internal logic in `convergence.rs` switches from
   `is_equality_constraint(g_l[i], g_u[i])` branching to direct c/d-block
   loops.
2. **filter.rs** ‚Äî `theta` (constraint violation) split into
   `theta_c + theta_d`. Equality contribution is `||g_c||`, inequality is
   `||max(d_l - d, 0) + max(d - d_u, 0)||`.
3. **kkt_aug.rs** ‚Äî `assemble_aug_kkt`, `build_outer_rhs`, `unpack_step`,
   `compute_sigma_s`. The 4-block KKT structure already matches; the
   change here is feeding `Jac_c` / `Jac_d` views (still combined-backed)
   instead of `(jac_rows, jac_cols, jac_vals) + partition.eq_pos`.
4. **kkt.rs** ‚Äî full-matrix variant; same migration as kkt_aug.
5. **restoration*.rs** ‚Äî `RestorationNlp` exposes its own combined `g`. We
   build the layout for the restoration NLP independently in
   `RestorationNlp::new`. Out of scope for this phase to split *its*
   storage ‚Äî the wrapper just needs to not assume the parent uses combined.
6. **ipm.rs** ‚Äî line search, watchdog, SOC, iter-0 probes, result
   unscaling. ~430 hits; tackle one bucket per commit.

Verification: `cargo test` green per commit. Spot-check `cargo run --bin
cutest_suite -- arki0003` to confirm iter-0 numbers unchanged (this is
Phase 2's invariant ‚Äî pure refactor).

#### Phase 2 status (2026-04-30)

Steps 1‚Äì5 are landed (commits 81c331e, 516aea1, 6fd01af, 826d131).
Filter (step 2) was a no-op: filter.rs is scalar-only (theta/phi);
the layout-aware migration happened in its callers in step 1.

Step 6 (ipm.rs internal sweep) is split:
* **Pass-throughs** (~250 reads) like `&state.g_l, &state.g_u` feeding
  `kkt::assemble_kkt` / `kkt_aug::*` / `compute_sigma_s` etc. are kept
  as-is ‚Äî those leaf functions are already layout-driven (steps 3, 4)
  so no rewire is needed until Phase 3 flips the storage shape.
* **Inline equality checks** at 12 sites of the form
  `state.g_l[i].is_finite() && state.g_u[i].is_finite() &&
   (state.g_l[i] - state.g_u[i]).abs() < 1e-14` use a 1e-14 threshold,
  whereas `ConstraintLayout` (and Ipopt) use 1e-15. Replacing them is
  a minor numeric change ‚Äî not a pure refactor ‚Äî and is deferred to a
  separate hypothesis-driven commit so any benchmark delta is
  attributable to that change in isolation.
* **Specific-row reads** (`state.y[i]`, `state.g[i]` at single indices
  inside line-search / SOC / iter-0 probes) are subsumed by Phase 3:
  flipping the storage means there is no combined `state.y` to read
  ‚Äî the migration becomes mechanical at that point.

The Phase 2 invariant ("no behaviour change, leaf modules layout-aware")
is met. 303 lib tests + all integration suites pass; the only failing
test (electrolyte_02_co2_water) is pre-existing on baseline.

### Phase 3 ‚Äî flip backing storage to split

After Phase 2, no caller of `state.g`/`g_l`/`g_u`/`s`/`y`/`g_scaling`
remains outside the accessors. We can now swap the fields:

```rust
// Before (today, ~src/ipm.rs:625-669):
pub g: Vec<f64>,
pub g_l: Vec<f64>,
pub g_u: Vec<f64>,
pub s: Vec<f64>,
pub y: Vec<f64>,
pub v_l: Vec<f64>,
pub v_u: Vec<f64>,
pub g_scaling: Vec<f64>,

// After:
pub g_c: Vec<f64>,
pub g_d: Vec<f64>,
pub d_l: Vec<f64>,
pub d_u: Vec<f64>,
pub s: Vec<f64>,    // size n_d only
pub y_c: Vec<f64>,
pub y_d: Vec<f64>,
pub v_l: Vec<f64>,  // size n_d
pub v_u: Vec<f64>,  // size n_d
pub c_scaling: Vec<f64>,
pub d_scaling: Vec<f64>,
```

Update `SolverState::new`, `ScaledProblem::constraint_bounds` /
`constraints` / `jacobian_values` to populate the split storage directly.
Accessors become trivial pass-throughs (and switch from returning `Vec` to
returning `&[f64]`).

Verification: `cargo test` green. `cargo run --bin cutest_suite --
arki0003` numerics unchanged. Result-unscaling code (`src/ipm.rs:8811,
8815`) updated to reconstruct user-ordered `g`/`y` via
`layout.c_to_combined`/`d_to_combined`.

#### Phase 3 status (2026-04-30)

The single Phase 3 storage-flip was decomposed into atomic sub-phases,
one per field family. Each sub-phase is committed in isolation with
303 lib tests + all integration suites green (only pre-existing
electrolyte_02_co2_water failure on baseline).

| Sub-phase | Field family | Storage | Commit |
|-----------|--------------|---------|--------|
| 3a | `g_scaling` ‚Üí `c_scaling`/`d_scaling` | n_c + n_d | 49b2cdc |
| 3b | `y` ‚Üí `y_c`/`y_d` | n_c + n_d | b22f59f |
| 3c | `dy` ‚Üí `dy_c`/`dy_d` | n_c + n_d | 0a22173 |
| 3d | `s`/`ds` (drop eq sentinel) | n_d | df9deb5 |
| 3e | `v_l`/`v_u`/`dv_l`/`dv_u` (drop eq zeros) | n_d | 371025f |
| 3f | add `c_rhs`/`d_l`/`d_u` alongside `g_l`/`g_u` | n_c + n_d each | a6d53b6 |

3f landed as additive (split fields populated alongside combined
`g_l`/`g_u`) rather than a full storage flip. Rationale: 125+ read
sites of `state.g_l` / `state.g_u` make a mass mechanical replace
high-risk for a refactor expected to be numerically inert. Bounds
don't change during solve, so the ~3¬∑m extra floats are one-shot
overhead. The native split fields exist on `SolverState` (`c_rhs`,
`d_l`, `d_u`) and Phase 4 readers can migrate incrementally to read
them once a layout-aware bound setter is added to the test-fixture
surface.

`state.g` (constraint values) was left as combined m-form scratch.
It is the natural target for the `eval_g` trait boundary, and
splitting it would require a transient `g_eval_buf` plus per-eval
projection ‚Äî overhead with no Phase 4 benefit (Phase 4 only consumes
the Jacobian split, not the constraint-value split).

**Phase 3 complete.** Phase 4 (Jacobian split) is unblocked.

### Phase 4 ‚Äî split Jacobian into Jac_c, Jac_d

Replace combined `(jac_rows, jac_cols, jac_vals)` with two sparse blocks.
`ScaledProblem::jacobian_values` populates them directly. `kkt_aug` no
longer consults `partition.eq_pos`/`ineq_pos` per triplet ‚Äî the split is
already done at storage time, matching Ipopt.

`ConstraintPartition` either becomes a no-op shim (kept temporarily for
test compatibility) or is deleted outright (rewriting the 11 tests).

Verification: `cargo test` green. Iter-0 dx/dy/ds on arki0003 must remain
*bit-identical* to Phase 3 output (refactor invariant). If they shift,
that's a Phase 4 bug, not the original gap.

#### Phase 4 status (2026-05-01)

Phase 4 was decomposed into two sub-phases:

- **Phase 4a (additive, commit 2842e39)** ‚Äî added `jac_c_*` / `jac_d_*`
  triplets and `jac_c_combined_idx` / `jac_d_combined_idx` index maps to
  `SolverState`. Structure built once in `SolverState::new` from the
  combined triplet via `layout.eq_pos` / `ineq_pos`; values refreshed
  each evaluate via `refresh_split_jac_vals` (one indirect read per

- **Phase 4b (consume + drop dispatch, commit c298e6a)** ‚Äî migrated
  the four `kkt_aug` entry points (`assemble_aug_kkt`,
  `aug_step_from_state`, `aug_step_from_state_mehrotra`, `aug_soc_solve_dx`,
  `aug_soc_solve_dx_factored`) and their three `ipm.rs` callers to

  centralised through `compute_j_t_y_split`. The augmented KKT path no
  longer touches the combined Jacobian.

Combined triplet still carried for non-augmented consumers
(`kkt::assemble_kkt`, restoration, sensitivity, convergence). Removing
it is out of scope for v0.8 ‚Äî it would require rewriting those modules
and is not on the hot path.

**Phase 4 complete.** 304 lib tests pass; integration baseline
unchanged from Phase 3 (only pre-existing
`electrolyte_02_co2_water` pH-assertion fails). Phase 5 (arki0003
trace verification) is unblocked.

### Phase 5 ‚Äî drop residual combined storage (`state.g` and `state.jac_*`)

Phase 4b cleared the augmented path of the combined Jacobian, but four
modules still consume it: `kkt::assemble_kkt` (legacy 3-block), the
restoration NLP wrapper, sensitivity, convergence (KKT residual / ls_y).
Combined `state.g` (size m) is similarly read by convergence checks,
restoration entry, scaling, and various probes.

Migration plan (mirrors Phases 3 / 4 sub-phasing):

- 5a. Add `state.g_c` (size n_c) + `state.g_d` (size n_d) as native
  storage; populate from the user trait via `layout.project_c/_d`
  immediately after `problem.constraints`. Keep `state.g` until
  consumers migrate.
- 5b. Migrate `kkt::assemble_kkt` to take split Jacobian + split y.
- 5c. Migrate convergence (`compute_kkt_residual`, `compute_avg_complementarity`,
  `ls_y_compute_initial_y`) to read `g_c`/`g_d` and `jac_c_*`/`jac_d_*`.
- 5d. Migrate restoration entry to project bounds/Jacobian into split form
  before constructing the inner NLP; restoration's internal NLP can
  remain combined (it's a different problem).
- 5e. Migrate sensitivity recovery to split form.
- 5f. Remove `state.g`, `state.jac_rows/cols/vals` from `SolverState`.

### Phase 6 ‚Äî compress bound multipliers `z_L`, `z_U` (introduce `Px_L`, `Px_U`)

Today: `state.z_l`, `state.z_u` are size `n` with `0` for unbounded sides.
Ipopt: `z_L` is size `n_x_l` (count of finite *lower* bounds), recovered
to full-n via the expansion matrix `Px_L` (`IpOrigIpoptNLP.hpp:197`).

Sub-phases:

- 6a. Add `state.bound_layout`: counts `n_x_l`, `n_x_u` plus index maps
  `x_l_to_full[k] ‚àà 0..n` and `full_to_x_l[i] ‚àà Option<usize>`. Same
  structure as `ConstraintLayout` but for variable bounds. Mirrors
  Ipopt's `Px_L_` ExpansionMatrix.
- 6b. Add native compressed `z_l_compressed: Vec<f64>` (size n_x_l),
  `z_u_compressed: Vec<f64>` (size n_x_u). Populate as projections of
  the combined `z_l`/`z_u` (additive).
- 6c. Migrate readers (sigma_x compute, rhs_z_l/u, dz_l/u recovery,
  frac-to-boundary, kappa_sigma reset, LOQO update) to consume the
  compressed form via the bound-layout maps.
- 6d. Remove combined `z_l`, `z_u` from `SolverState`.

### Phase 7 ‚Äî compress variable bounds `x_L`, `x_U`

Today: `state.x_l`, `state.x_u` size `n`, with `¬±‚àû` for unbounded.
Ipopt: `x_L` size `n_x_l`, `x_U` size `n_x_u` (only finite entries).

Sub-phases parallel Phase 6: native compressed storage, migrate readers,
drop combined.

### Phase 8 ‚Äî compress slack-bound multipliers `v_L`, `v_U` (introduce `Pd_L`, `Pd_U`)

Today: `state.v_l`, `state.v_u` size `n_d` with `0` for unbounded sides.
Ipopt: `v_L` size `n_d_l`, `v_U` size `n_d_u` recovered via `Pd_L`,
`Pd_U` (`IpOrigIpoptNLP.hpp:241,263`).

Sub-phases mirror Phase 6 but for the d-block.

### Phase 9 ‚Äî compress slack-bound storage `d_L`, `d_U`

Today: `state.d_l`, `state.d_u` size `n_d`, `¬±‚àû` for unbounded sides.
Ipopt: `d_L` size `n_d_l`, `d_U` size `n_d_u`.

Sub-phases parallel Phase 8.

### Phase 10 ‚Äî physical row reorder at the IPM boundary

Ipopt's `TNLPAdapter` returns the user-supplied `g(x)` and `jac_g`, then
*physically* projects into split form using `c_map`/`d_map`. After
TNLPAdapter, no combined m-vector exists.

ripopt today preserves the user's row order in the combined-form fields
that survived earlier phases. After Phases 5‚Äì9, the only remaining user-
row-order surface is the `NlpProblem` trait itself (`fn constraints`,
`fn jacobian_*`). This phase formalises the boundary:

- 10a. Add a thin internal "TNLPAdapter equivalent" that wraps the user's
  `NlpProblem` and exposes only split-form `c(x)`, `d(x)`, `Jac_c`, `Jac_d`.
- 10b. `SolverState::new` calls this adapter; never sees combined again.
- 10c. The `NlpProblem` trait surface remains user-facing combined (no
  user-visible change).

### Phase 11 ‚Äî fixed-variable strategy (Ipopt `MAKE_PARAMETER`)

Today: ripopt relaxes fixed-variable bounds (`relax_fixed_variable_bounds`)
and keeps fixed vars in `x` at full size n.
Ipopt: `MAKE_PARAMETER` mode physically removes fixed vars from `x`,
sizing internal `x` to `n_x_var = n_full ‚àí n_x_fixed`. Recovery via
`P_x_full_x_` (`IpTNLPAdapter.hpp:399`).

Sub-phases:

- 11a. Add `state.x_layout`: counts `n_full`, `n_x_var`, maps.
- 11b. Compress `x`, `x_l`, `x_u`, `z_l`, `z_u`, `grad_f`, gradient,
  Hessian rows/cols, Jacobian cols.
- 11c. Switch fixed-variable handling from "relax bounds" to "exclude from x".
- 11d. Drop `relax_fixed_variable_bounds`.

### Phase 12 ‚Äî verify against Ipopt arki0003 trace

(Was old Phase 5.) With every storage form Ipopt-aligned, the iter-0
probe at `src/ipm.rs:5743-5832` becomes a direct, element-by-element
comparison against `/tmp/arki0003_ipopt_trace.txt`. No projection,
no zero-padding aliasing, no row-permutation ambiguity remains. If the
6.16% dx[1753] / 13% dy[e2253] gaps survive, root cause is downstream
of storage layout (Lagrangian sign, œÉ scaling, H-block).

## Out of scope (none ‚Äî full Ipopt alignment is the target)

The earlier "out of scope" list (Px_L/Px_U/Pd_L/Pd_U compression,
TNLPAdapter row reorder, fixed-variable strategy) has been promoted
into Phases 6‚Äì11. The only remaining user-facing combined surface is
the `NlpProblem` trait itself, which is intentional ‚Äî it matches
Ipopt's TNLP user contract.
