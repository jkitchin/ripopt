# v0.8 IPM Algorithmic Improvement Plan

Part of the v0.8 release. The first half of v0.8 swapped rmumps for feral
as the default sparse linear solver (already merged on
`v0.8/feral-default`). This second half closes ripopt's algorithmic gap
with Ipopt 3.14 on the CUTEst benchmark, built from three parallel
reviews:

1. **Ipopt-expert catalog** — every IPM mechanism in `ref/Ipopt/` with file:line
   citations, formulas, and defaults (12 algorithmic categories).
2. **ripopt implementation catalog** — what `src/ipm.rs`, `src/kkt.rs`,
   `src/filter.rs`, `src/restoration*.rs`, `src/sqp.rs`, `src/lbfgs.rs`,
   `src/augmented_lagrangian.rs`, `src/convergence.rs`, `src/options.rs`
   actually do today.
3. **CUTEst failure mining** — clusters of the 170 ripopt non-Optimal results
   (`benchmarks/cutest/results.json`, v0.8.0-dev, 727 problems × 2 solvers).

The full agent transcripts that produced these catalogs are not committed —
re-running them is cheap (~6 min wall, ~150k tokens). Their findings are
distilled below.

---

## Why these IPM changes belong in v0.8

The first half of v0.8 replaced rmumps with feral. On the HS benchmark,
v0.8.0 with feral is **byte-for-byte identical** to v0.7.1 with rmumps on
correctness (118/120 Optimal, same failures HS116 / HS374). Geomean timing
differs by 4.5%. **The linear solver is not the bottleneck.** The
remaining ~13% pass-rate gap to a hypothetical perfect solver lives in the
IPM itself — barrier strategy, line search, restoration, fallback
dispatch. Shipping v0.8.0 without addressing it would leave the headline
"new default solver" change on the table without the user-visible
correctness wins that motivated the swap in the first place.

CUTEst v0.8.0-dev: **ripopt 557 / 727 (76.6%)** vs **ipopt 556 / 727 (76.5%)**.
Aggregate parity hides directional asymmetry: 40 problems where only ipopt
solves, 41 where only ripopt solves. The 40-problem gap is what this plan
attacks.

### Important: the original "T1.4 status fix" was wrong

The CUTEst failure-mining agent first reported "21 problems with all-zero
residuals labeled LocalInfeasibility." Re-running the filter against
`constraint_violation` (not just `final_primal_inf`) shows only **YFITNE**
has a truly zero residual; the other 74 have nonzero `constraint_violation`
(BARDNE: cv=0.082, BOXBOD: cv=23.4, BROWNDENE: cv=178.97). These are
genuinely locally infeasible NE/curve-fit problems where ripopt converges
to a least-squares stationary point with nonzero residual. Ipopt rejects
them with `IpoptStatus(-10)`. **No status-classification fix is justified.**
This plan does not include T1.4.

## Tier 1 — high-leverage, low-complexity

Each task is independently mergeable. Each lists: scope, estimated effort,

### T1.1 `bound_relax_factor` at problem setup

**Estimated effort**: 0.5 day.
**Targets failure cluster**: (b) dual-residual blowup, plus part of (a).
**Expected status flips**: ~5–10 (AC-OPF + near-bound init).

**Mechanism**:
At problem setup, after loading variable bounds and before `bound_push`,
relax each finite bound outward by the standard Ipopt formula:

```
For each i with x_L[i] finite:
    relax = bound_relax_factor * max(|x_L[i]|, 1.0)
    x_L[i] -= max(EPS, relax)             # cap by constr_viol_tol if needed
For each i with x_U[i] finite:
    relax = bound_relax_factor * max(|x_U[i]|, 1.0)
    x_U[i] += max(EPS, relax)
```

Default: `bound_relax_factor = 1e-8` (Ipopt default). 0.0 to disable.
Apply BEFORE `relax_fixed_variable_bounds` (which handles x_L == x_U).

Does NOT relax constraint bounds g_L, g_U in the first pass — that's a
separate question to revisit if AC-OPF still fails after this.

**Files**:
- `src/options.rs` — add `pub bound_relax_factor: f64,` near `bound_push`;
  default `1e-8` in `Default::default()`.
- `src/ipm.rs:8917` — add `apply_bound_relax_factor` near
  `relax_fixed_variable_bounds`. Call it from the init code at
  `src/ipm.rs:802-810` before `relax_fixed_variable_bounds`.

**Narrow test**:
```
RESULTS_FILE=/tmp/cutest_t1_1.json \
cargo run --bin cutest_suite --features cutest,ipopt-native --release -- \
    ACOPP30 ACOPR14 ACOPR30 BIGGSC4 HS83 HS84 HS118 HATFLDH SWOPF SNAKE \
    HIMMELBI BATCH CORE1 FEEDLOC HAIFAM
```
Compare against the same problems in
`benchmarks/cutest/results.json` (v0.8.0-dev baseline).

**Acceptance**: ≥ 3 status flips Optimal-ward; 0 regressions on other
problems in the slice.

### T1.2 `slack_move` runtime slack adjustment

**Estimated effort**: 1 day.
**Targets failure cluster**: (a) late-stage NumericalError.
**Expected status flips**: ~10–15.

**Mechanism**:
Ipopt nudges slacks outward when they fall below `slack_move = ε^0.75 ≈
1.83e-12` mid-solve. The check runs after every accepted iterate and
nudges the corresponding *bound* outward (not the slack directly):

```
After accepted step:
    For each i with x_L[i] finite:
        s_l = x[i] - x_L[i]
        if s_l < slack_move:
            new_s = min(max(mu / z_l[i], slack_move),
                        slack_move * max(1.0, |x_L[i]|))
            x_L[i] -= (new_s - s_l)         # bound moves left, slack widens
    Same for x_U, mirror.
```

Track `state.adjusted_slacks_count`; surface in iteration trace as `'a'`
(matching Ipopt's info char).

**Files**:
- `src/options.rs` — `pub slack_move: f64,` default `f64::EPSILON.powf(0.75)`.
- `src/ipm.rs` — new `apply_slack_move` called from accept-trial-point
  routine. Increment counter on every adjustment.

**Narrow test**:
```
HYDC20LS OET7 OET4 OET6 LINSPANH DECONVBNE CERI651ALS CERI651BLS \
    MISTAKE ALLINITA ALLINITC NET1 SPANHYD DUALC8 FBRAIN3LS HATFLDFL \
    PALMER7E HIMMELBI DEMBO7 DMN15B0 DMN15B1 DMN37B0
```

**Acceptance**: ≥ 5 status flips Optimal-ward; 0 regressions in the slice.

### T1.3 `kappa_d` damping for one-sided bounds

**Estimated effort**: 1 day.
**Targets failure cluster**: one-sided-bound stalls.
**Expected status flips**: ~3–5 (CRESC50, OET3, OET6, FEEDLOC, SNAKE).

**Mechanism**:
For variables with exactly one finite bound (XOR), add a linear damping
term to the barrier objective that penalizes drift toward infinity:

```
For each i with exactly one of x_L[i], x_U[i] finite:
    damping = kappa_d * mu * slack_oneside[i]    # slack_oneside = x[i] - x_L[i] or x_U[i] - x[i]
    Add to barrier objective.
    Add kappa_d * mu to gradient_barrier_objective[i] (with appropriate sign).
```

Default `kappa_d = 1e-5`. Without this term, the barrier objective is
unbounded below for one-sided-bound variables (Ipopt's CalcQty:155-160).

**Files**:
- `src/options.rs` — `pub kappa_d: f64,` default `1e-5`.
- `src/ipm.rs` — modify `barrier_objective` and `gradient_barrier_objective`
  to add the damping term for one-sided variables. Identify them once at
  setup; cache in state.

**Narrow test**:
```
CRESC50 OET3 OET4 OET6 FEEDLOC SNAKE QPCBLEND QPNBLEND HS83 HS84 \
    HS118 HATFLDH BIGGSC4
```

**Acceptance**: ≥ 3 status flips; 0 regressions.

## Tier 2 — moderate-complexity

### T2.1 `RestoreAcceptablePoint` mechanism

**Estimated effort**: 2–3 days.
**Targets failure cluster**: (a) and (c).
**Expected status flips**: ~5.

**Mechanism**:
1. Add `state.acceptable_iterate: Option<IterateSnapshot>` capturing
   `(x, s, y, z_l, z_u, mu, filter_clone, iteration)`.
2. On every iteration where the convergence check returns `Acceptable`
   (looser than Converged but still passes the soft tolerance ladder),
   call `store_acceptable_iterate(state)` to overwrite the snapshot.
3. Before triggering full restoration, check
   `pinf < 1e-2 * tol AND constr_viol_max < 1e-1 * constr_viol_tol`.
   If true, restore the snapshot and return `Acceptable` (matches Ipopt's
   `STOP_AT_ACCEPTABLE_POINT`). If no snapshot exists, fall through to
   restoration as today.

**Why this matters**: Ipopt observed that triggering restoration on a
near-feasible iterate produces an ill-defined resto NLP (constraints
already nearly satisfied → no work for slacks p, n → singular). ripopt
currently has no escape hatch.

**Files**:
- `src/ipm.rs` — `IterateSnapshot` struct, `store_acceptable_iterate`,
  `try_restore_acceptable_iterate` near the restoration trigger.
- `src/convergence.rs` — pass the `Acceptable` signal back up so the IPM
  loop can call `store_acceptable_iterate`.

**Narrow test**:
```
PFIT2 TRO6X2 FLETCHER ALLINITA ALLINITC HS83 HS84 SPIRAL POWELLSQ
```

**Acceptance**: ≥ 3 status flips; verify the existing RestorationFailed
problem (the 1 in `benchmarks/cutest/results.json`) doesn't regress.

### T2.2 Square-problem auto-tweaks

**Estimated effort**: 0.5 day.
**Targets failure cluster**: (b) AC-OPF.
**Expected status flips**: ~2–3.

**Mechanism**:
At setup, set `state.is_square = (m == n) || (m_eq == n)`. When square:
- Set `kappa_resto = 0` in `src/restoration.rs` (already a value, gate on
  `is_square`).
- Disable the `early_stall_timeout` infeasibility-declaration path
  (square problems can have legitimately slow first iterations).

Mirrors `IpBacktrackingLineSearch.cpp:276-280`.

**Files**:
- `src/ipm.rs` — `state.is_square` flag.
- `src/restoration.rs` — gate `kappa_resto` on `is_square`.

**Narrow test**: ACOPP30, ACOPR14, ACOPR30, plus a 5-problem control set
of non-square problems to verify no regression.

**Acceptance**: ≥ 1 AC-OPF status flip; 0 regressions on control set.

## Tier 3 — broader hardening

### T3.1 `mu_allow_fast_monotone_decrease`

**Estimated effort**: 0.5 day.

In monotone branch: after a μ decrease, if `barrier_err ≤
barrier_tol_factor * mu_new`, decrease μ again in the same iteration.
Cap at `max_consecutive_decreases = 4`. Mirrors `IpMonotoneMuUpdate.cpp:79`.

### T3.2 NaN/Inf wrapper on every NLP eval

**Estimated effort**: 1 day.

Wrap each call to `problem.objective`, `gradient`, `constraints`,
`jacobian_values`, `hessian_values` with a finite-number check. On NaN/Inf,
return `SolverError::InvalidNumber` and route to `NumericalBreakdown` in
the failure dispatcher. Today some checks are scattered; this consolidates
into one wrapper layer.

### T3.3 Quality-function oracle: golden section

**Estimated effort**: 2 days. (Optional.)

Replace ripopt's discrete-sampling quality oracle with a proper
golden-section minimization over `[1e-6, 1e2]` with tolerance `1e-2` and
max 8 sections. Matches `IpQualityFunctionMuOracle.cpp`.

### T3.4 Free-mode τ formula alignment

**Estimated effort**: 0.5 day.

Switch `τ = max(τ_min, 1 − (θ + du + dc))` to Ipopt's `τ = max(τ_min,
1 − E_μ)` where `E_μ` is the unified KKT-error norm.

## Validation strategy

For each task:

1. **Branch off the current `v0.8/feral-default` branch** for each task,
   e.g. `v0.8/ipm-bound-relax-factor`. Merge sequentially back into
   `v0.8/feral-default` so the v0.8.0 tag captures all IPM improvements.
2. **Implement, compile clean** under all three feature combos:
   - `cargo build` (default = feral + faer)
   - `cargo build --no-default-features --features "rmumps faer"`
   - `cargo build --no-default-features` (dense fallback)
3. **Run unit tests**: `cargo test` should stay 292/292 green.
4. **Run narrow CUTEst slice** (per-task list above).
5. **Diff against v0.8.0-dev (feral-only) baseline**: count status flips
   Optimal-ward and any regressions. Save a snapshot of
   `benchmarks/cutest/results.json` from the current
   `v0.8/feral-default` HEAD as `benchmarks/cutest/results_v0.8.0-dev_feral_only.json`
   so each task has the right baseline to compare against.
6. **Merge gate**: ≥ N flips per task (specified above), 0 regressions.
7. **After Tier 1 complete**: run full CUTEst sweep before tagging v0.8.0;
   the post-IPM-improvements `BENCHMARK_REPORT_v0.8.0.json` becomes the
   release baseline.

For Tier 2 / Tier 3: same workflow, narrower target slice each time.
